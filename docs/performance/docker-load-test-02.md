## 검증 목적

2차 테스트는 1차 테스트 이후의 흐름을 이어받아 부하 조건과 확인 범위를 넓히는 검증입니다. 초기 환경 구성에서 확인한 내용을 바탕으로 서버 처리량, 응답 지연, 오류 발생 조건을 더 구체적으로 점검하는 데 목적이 있습니다.

## 중점 정리 항목

- 1차 테스트 결과를 기준으로 테스트 조건 재확인
- Docker 기반 실행 상태에서 반복 요청과 동시 요청 처리 흐름 확인
- API 응답 지연, DB 연결, 서버 로그의 이상 징후 확인
- 테스트 중 확인된 병목 후보를 운영 코드와 배포 설정 관점에서 분류
- 다음 회차에서 재검증해야 할 항목 정리

## 저장소 공개 방식

2차 테스트는 성능 개선의 중간 기록으로 관리합니다. 저장소에는 테스트 결론과 점검 기준을 남기고, 원문 블로그는 상세 과정과 이미지 자료의 출처로 연결합니다. 이 방식은 GitHub 문서가 지나치게 길어지는 것을 막고, 테스트 증빙은 외부 링크로 확인할 수 있게 합니다.

## 후속 확인 항목

- 동일한 테스트 조건에서 코드 변경 전후 응답 차이를 비교합니다.
- DB 연결 제한, 요청 제한, hCaptcha, 인증 미들웨어가 부하 상황에서 의도대로 동작하는지 확인합니다.
- Socket.IO, 게시판, 문제 풀이, 회식맵처럼 사용자가 동시에 접근할 수 있는 기능의 서버 로그를 분리해 확인합니다.
